# **Git 提交规范**

### **一、为什么要规范 Git 提交？**

在任何软件项目中，无论是个人项目还是团队协作，清晰、一致的 Git 提交历史都是至关重要的资产。它不仅是代码变更的记录，更是项目演进的“故事书”。一个规范的提交历史能够：

- **提高代码审查效率**：审查者可以快速理解每次变更的目的和背景。
- **简化问题追溯过程**：当出现 Bug 时，可以迅速定位到相关的代码提交。
- **帮助新成员快速融入项目**：通过阅读提交历史，新人可以更快地了解项目的发展脉络和技术决策。

本规范基于业界广泛应用的 **Angular 提交规范（Conventional Commits）**，旨在提供一套简单、清晰且易于维护的 Git 提交信息格式，使提交历史更具可读性，助力高效的开发流程。

---

### **二、规范格式**

每一次的 Git 提交信息都应严格遵循以下格式：

```text
<type>(<scope>): <subject>

[可选的正文]

[可选的脚注]
```

---

### **三、核心组成部分**

1. **类型（Type）**：**（必填）** 用于说明此次提交的性质。它决定了这次提交将出现在哪个变更日志区域。类型必须是预定义关键字之一（见下表）。
2. **范围（Scope）**：**（可选）** 用于描述本次提交影响的代码范围。这可以是模块名、组件名、文件名或功能区域，有助于快速定位变更。例如 `readme`、`i2c`、`app` 等。
3. **主题（Subject）**：**（必填）** 对本次提交的简短描述。

   **规则：**

   - 使用祈使句现在时态，例如使用“修复”而不是“修复了”。
   - 如用英文，首字母不需要大写。
   - 结尾不加句号。
   - 长度不应超过 50 个字符。
4. **正文（Body）**：**（可选）** 如果提交的变更比较复杂，不足以在主题中说明清楚，则应使用正文部分。

   **规则：**

   - 与主题之间必须有一个空行。
   - 详细解释**为什么**要进行这次变更，而不仅仅是**做了什么**。
   - 可以包含变更前后的行为对比。
   - 每行长度建议不超过 72 个字符，以便在终端中阅读。
5. **脚注（Footer）**：**（可选）** 用于提供额外的信息。

   **规则：**

   - 与正文之间必须有一个空行。
   - 可用于说明关联 Issue、兼容性变更等附加信息。
6. **不兼容变更**：如果当前代码与上一个版本不兼容，脚注部分必须以 **`BREAKING CHANGE:`** 开头，后面是对变更的描述、理由和迁移方法。
7. **关联 Issue**：可以关闭一个或多个 Issue，例如 `Closes #123, #456`。

---

### **四、推荐类型（Type）**

| **类型** | **描述** |
| --- | --- |
| **feat** | 引入新功能。 |
| **fix** | 修复一个 Bug。 |
| **docs** | 添加或更新文档。 |
| **style** | 改进代码格式或风格，不影响代码含义。 |
| **refactor** | 代码重构，既不是修复 Bug 也不是添加新功能。 |
| **perf** | 提升性能的代码更改。 |
| **test** | 添加、更新或修正测试。 |
| **build** | 影响构建系统或外部依赖的更改。 |
| **ci** | 添加或更新 CI 配置、流程或脚本。 |
| **chore** | 其他不修改业务源码或测试代码的提交。 |
| **revert** | 撤销之前的某次提交。 |

### **五、提交示例**

#### **示例 1：修复底层驱动 Bug**

此示例修复了一个在高波特率下，因 CPU 处理不及时导致 UART 接收字节丢失的问题。`scope` 指向硬件模块 `uart`。

```text
fix(uart): 修正高波特率下接收字节丢失的问题

原有的轮询接收方式在波特率高于 115200 时，CPU 无法及时读取硬件缓冲区数据，
导致缓冲区溢出和字节丢失。

本次提交将接收方式从轮询改为 DMA 模式。通过 DMA 在后台自动将数据从 UART 外设
传输到内存，彻底解决了高负载下的数据丢失问题，并显著降低了 CPU 占用率。
```

#### **示例 2：添加新功能**

此示例为产品添加了 OTA（空中升级）功能，并关联了某个 `issue`。`scope` 指向具体业务功能 `ota`，因为它涉及多个模块（如 `wifi`、`flash`）。

```text
feat(ota): 实现固件下载与验证功能

增加了空中升级（OTA）的核心逻辑。设备现在可以连接到指定服务器，下载新的固件
二进制文件，并在重启前验证其签名。

Closes #42
```

#### **示例 3：代码重构与不兼容变更**

此示例重构了硬件抽象层 `hal`，并引入了一个不兼容的 API 变更。注意 `BREAKING CHANGE` 脚注的用法。

```text
refactor(hal): 统一所有通道的 ADC 接口

通过创建一个统一的 `hal_adc_read` 函数来代替为每个通道编写独立函数，改进了
ADC 的硬件抽象层。这简化了 API 并减少了代码冗余。

BREAKING CHANGE: 函数 `hal_adc_read_channel_1()` 和 `hal_adc_read_channel_2()`
已被移除。现在所有 ADC 读数都必须使用 `hal_adc_read(channel_num)` 来执行。
```

#### **示例 4：更新构建脚本**

此示例修改了 `CMake` 配置文件，`scope` 为 `build`，属于既不影响源码也不影响测试的 `chore` 类型。

```text
chore(build): 为硬件 FPU 添加编译器标志

启用了 `-mfloat-abi=hard` 编译器标志以利用硬件浮点单元（FPU）。这显著提高了
传感器融合算法中浮点计算的性能。
```

#### **示例 5：更新文档**

此示例更新了项目的 `README.md` 文件，`scope` 为 `readme`，类型为 `docs`。

```text
docs(readme): 更新项目介绍和使用说明

根据最新的硬件版本和软件功能，更新了 README 文件中的项目描述、功能列表和
快速上手指南，使其与当前项目状态保持一致。
```
