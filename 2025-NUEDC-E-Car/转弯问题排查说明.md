# 转弯后无法回到直线循迹模式排查说明

## 1. 问题现象

当前 TI 小车项目在执行左转后，表现为不能稳定回到正常直线循迹状态。实际运行中更可能出现的现象是：

- 小车已经完成了一次转弯动作。
- 状态机短暂回到循迹态。
- 由于转弯触发条件仍然成立，下一拍又重新进入转弯态。
- 从外部表现看，就像“转弯后出不来”或者“转完后还在继续拐”。

本次排查仅基于现有代码逻辑分析，未改动代码。

## 2. 排查范围

本次重点检查了以下文件：

- [empty.c](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c)
- [motor.c](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\user_driver\motor.c)

其中，转弯与循迹状态机主逻辑位于 `empty.c` 的 `Track_Task()` 中，速度闭环位于 `motor.c` 的 `Motor_Control_Loop()` 中。

## 3. 当前状态机结构

`empty.c` 中定义了 3 个状态：

- `STATE_TRACKING`
- `STATE_TURN_LEFT_BLIND`
- `STATE_TURN_LEFT_ROTATE`

对应代码位置：

- [empty.c:328](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:328)
- [empty.c:334](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:334)

当前逻辑流程可以概括为：

1. 在 `STATE_TRACKING` 下执行正常循迹。
2. 满足左转触发条件后，切换到 `STATE_TURN_LEFT_BLIND`。
3. 盲走一段时间后，切换到 `STATE_TURN_LEFT_ROTATE`。
4. 在旋转状态下，只要检测到某些传感器见线，就立即切回 `STATE_TRACKING`。

对应核心代码位于：

## 4. 关键代码分析

### 4.1 左转进入条件偏宽

在 `STATE_TRACKING` 中，进入左转状态的条件是：

```c
if (s.L3 == 1 && s.R3 == 0)
```

位置：

- [empty.c:349](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:349)

这意味着只要最左侧传感器 `L3` 检到线、最右侧 `R3` 没检到线，就会直接进入左转流程。

这个判定存在两个问题：

- 条件过于单一，只依赖 `L3` 和 `R3`，没有结合中间传感器判断当前是否真的是“路口左转触发”。
- 转弯结束后，车身姿态还没完全摆正时，`L3` 很可能仍然压在线上，此时刚回到 `STATE_TRACKING`，又会再次满足该条件。

这会导致状态机出现反复进入转弯态的风险。

### 4.2 出弯条件过早

在 `STATE_TURN_LEFT_ROTATE` 中，退出转弯状态的条件是：

```c
if (s.C == 1 || s.L1 == 1 || s.R1 == 1 || s.R2 == 1 || s.R3 == 1 || s.L2 == 1)
```

位置：

- [empty.c:383](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:383)

只要 `C/L1/L2/R1/R2/R3` 任意一个传感器见线，就立即执行：

```c
current_state = STATE_TRACKING;
last_error = 0;
```

位置：

- [empty.c:384](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:384)
- [empty.c:385](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:385)

这个退出条件同样偏宽，原因是：

- 只要侧边传感器短暂扫到线，就可能提前退出旋转。
- 退出时没有要求“小车已经回正并重新稳定压住主线”。
- 原本用于限制最短转向时间的判断被注释掉了。

被注释的代码位置：

- [empty.c:382](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:382)
- [empty.c:397](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:397)

这会导致一种典型情况：车还没真正转正，只是某个传感器先扫到线，就过早回到循迹态。

### 4.3 缺少“再次触发保护”

当前状态机没有看到以下保护机制：

- 转弯完成后等待 `L3` 释放，再允许下一次左转触发。
- 从 `STATE_TURN_LEFT_ROTATE` 切回 `STATE_TRACKING` 后，设置一个短暂的禁止再触发窗口。
- 使用“边沿触发”而不是“电平触发”识别左转。

这意味着当前设计是：

- 只要条件成立，就可立即再次进入左转。

因此即使代码已经执行了：

```c
current_state = STATE_TRACKING;
```

下一次定时中断进入 `Track_Task()` 时，仍可能因为 `L3` 还有效，再次执行：

```c
current_state = STATE_TURN_LEFT_BLIND;
```

对应位置：

- [empty.c:350](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:350)

这就是“状态看起来回到了循迹，但运行表现没有回到直线循迹”的主要原因候选。

## 5. 最可能的状态机异常链路

结合当前代码，最可能出现的实际状态切换链如下：

1. 小车在 `STATE_TRACKING` 正常循迹。
2. `L3 == 1 && R3 == 0`，进入 `STATE_TURN_LEFT_BLIND`。
3. 盲走结束后进入 `STATE_TURN_LEFT_ROTATE`。
4. 某个侧边或中间传感器先扫到线，满足退出条件。
5. 程序把状态切回 `STATE_TRACKING`。
6. 但此时车头仍偏左，`L3` 依旧可能为 1。
7. 下一拍再次满足左转进入条件。
8. 程序再次切回 `STATE_TURN_LEFT_BLIND`。

从逻辑上看，这会形成下面的循环：

`TRACKING -> TURN_LEFT_BLIND -> TURN_LEFT_ROTATE -> TRACKING -> TURN_LEFT_BLIND ...`

因此从现象上就会看到“转弯后无法恢复直线循迹”。

## 6. 对电机闭环的判断

目前看，主因更偏向状态机，而不是 PID 速度闭环。

原因如下：

- `motor.c` 中的 `Motor_Control_Loop()` 主要负责根据目标速度做双轮闭环控制。
- 在两轮目标速度相等时，还做了同步补偿。
- 这些逻辑会影响转弯动作快慢、姿态是否平顺，但不会单独造成“状态已经回循迹却又立刻继续转弯”的问题。

参考位置：

- [motor.c:130](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\user_driver\motor.c:130)
- [motor.c:141](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\user_driver\motor.c:141)

需要说明的是，转弯速度参数确实会放大状态机问题：

- 当前盲走结束后的速度设置为 `Car_SetSpeed(-1, 30)`。
- 这会让左轮轻微反转、右轮前进，形成原地偏左旋转。

位置：

- [empty.c:377](E:\Projects\2025-NUEDC-E\2025-NUEDC-E-Car\test1\empty.c:377)

如果旋转过快，传感器扫线会更突兀，更容易触发“提前退出”与“立即重入”的连锁问题。但这仍然是放大因素，不是最核心根因。

## 7. 当前结论

本次排查的核心结论如下：

1. 问题最可能出在 `Track_Task()` 的左转状态机切换逻辑，而不是底层电机闭环本身。
2. 左转进入条件过宽，导致出弯后容易再次被判定为左转。
3. 左转退出条件过早，导致车辆尚未完全对正主线就回到循迹态。
4. 代码缺少“转弯完成后的再触发保护”，使得状态机容易在循迹态和转弯态之间反复抖动。

综合判断，“转弯后不能回到直线循迹模式”本质上更像是：

- 实际上回到了 `STATE_TRACKING`
- 但马上又再次进入左转流程

因此从现象上看像“回不去”，从代码层看更像“回去后立刻又被拉出来”。

## 8. 建议的验证方式

在不立即改代码的前提下，建议先验证以下几点：

1. 在 OLED 或串口中实时输出 `current_state`。
2. 同时输出 `L3/L2/L1/C/R1/R2/R3` 当前值。
3. 重点观察一次完整左转后的 100 到 300 ms 内，状态是否经历：
   `STATE_TURN_LEFT_ROTATE -> STATE_TRACKING -> STATE_TURN_LEFT_BLIND`
4. 重点观察回到 `STATE_TRACKING` 时，`L3` 是否仍为 1。

如果以上现象成立，就基本可以确认本次判断。

## 8.1 推荐打印的调试内容

建议优先使用串口打印，原因是 OLED 刷新频率较低、显示内容有限，不适合抓瞬时状态切换。OLED 更适合只看摘要变量，串口更适合看完整时序。

推荐打印以下变量：

1. `sys_tick`
2. `current_state`
3. `left_turn_count`
4. `target_left_turns`
5. `last_error`
6. `target_speed_L`
7. `target_speed_R`
8. `current_speed_L`
9. `current_speed_R`
10. `L3/L2/L1/C/R1/R2/R3`

其中最关键的是：

1. `sys_tick`
2. `current_state`
3. `L3/L2/L1/C/R1/R2/R3`
4. `target_speed_L/target_speed_R`

## 8.2 建议重点打印的位置

不建议在每次定时中断里无条件全量打印，否则串口会严重刷屏，影响实时性。更适合的做法是只在以下时机打印：

1. `STATE_TRACKING` 进入 `STATE_TURN_LEFT_BLIND` 时打印一次。
2. `STATE_TURN_LEFT_BLIND` 进入 `STATE_TURN_LEFT_ROTATE` 时打印一次。
3. `STATE_TURN_LEFT_ROTATE` 退出到 `STATE_TRACKING` 时打印一次。
4. 若回到 `STATE_TRACKING` 后又再次进入左转，再打印一次。
5. 如果怀疑出弯后的 100 到 300 ms 内有状态抖动，可以临时在这段窗口内每隔若干 tick 打印一次。

这样能直接看到是不是发生了“刚回循迹又再次入弯”。

## 8.3 推荐的打印格式

建议每条日志至少包含时间戳、状态、传感器值、目标速度。

推荐格式一，适合打印状态切换事件：

```c
printf("[T=%lu] STATE=%d L3=%d L2=%d L1=%d C=%d R1=%d R2=%d R3=%d TarL=%d TarR=%d CurL=%d CurR=%d Err=%d\r\n",
       sys_tick,
       current_state,
       s.L3, s.L2, s.L1, s.C, s.R1, s.R2, s.R3,
       target_speed_L, target_speed_R,
       current_speed_L, current_speed_R,
       last_error);
```

推荐格式二，适合单独打印“状态切换原因”：

```c
printf("[T=%lu] TRACKING->BLIND, L3=%d R3=%d\r\n", sys_tick, s.L3, s.R3);
printf("[T=%lu] BLIND->ROTATE, TarL=%d TarR=%d\r\n", sys_tick, target_speed_L, target_speed_R);
printf("[T=%lu] ROTATE->TRACKING, L3=%d L2=%d L1=%d C=%d R1=%d R2=%d R3=%d\r\n",
       sys_tick, s.L3, s.L2, s.L1, s.C, s.R1, s.R2, s.R3);
```

如果串口带宽不够，可以先缩减为：

```c
printf("[T=%lu] S=%d Sen=%d%d%d%d%d%d%d Tar=%d,%d\r\n",
       sys_tick, current_state,
       s.L3, s.L2, s.L1, s.C, s.R1, s.R2, s.R3,
       target_speed_L, target_speed_R);
```

## 8.4 推荐的 OLED 调试内容

如果要用 OLED，建议只显示最关键的 4 组信息：

1. 当前状态：`STATE`
2. 传感器摘要：`L3 L2 L1 C R1 R2 R3`
3. 目标速度：`TarL TarR`
4. 当前速度：`CurL CurR`

推荐显示方式如下：

```text
STATE: 0/1/2
SEN: 1011000
TAR: L-1 R30
CUR: Lxx Rxx
```

如果 OLED 空间不够，优先保留：

1. `STATE`
2. `SEN`
3. `TarL/TarR`

## 8.5 推荐的判读方法

调试时，建议重点观察以下几种日志组合。

### 情况 1：刚出弯就再次入弯

如果日志出现下面这种顺序：

```text
ROTATE->TRACKING
TRACKING->BLIND
```

并且两条日志时间间隔很短，例如只有几个 `sys_tick`，说明当前问题基本就是状态机重入。

这时再看第二条日志里的传感器值，如果 `L3` 仍为 1，基本可以确认是左转触发条件未释放。

### 情况 2：旋转还没完成就退出

如果在 `ROTATE->TRACKING` 这一条日志中看到：

- `C` 没有稳定为 1；
- 只是 `L2`、`R2`、`R3` 这类侧向传感器先触发；

那就说明旋转退出条件偏早，车还没真正回到主线中心。

### 情况 3：状态没问题，但车身姿态不对

如果日志显示：

- 状态没有反复切换；
- `STATE_TRACKING` 也保持住了；
- 但车还是走偏或继续甩尾；

这时才应该更多考虑：

1. `Car_SetSpeed(-1, 30)` 的转弯速度是否过激；
2. 左右轮 PID 是否存在响应不对称；
3. 机械结构或编码器测量是否有偏差。

也就是说，先用日志排除状态机重入，再考虑运动控制参数。

## 8.6 最小化调试建议

如果你只想先加最少的打印，我建议至少加下面 3 条：

1. `TRACKING -> TURN_LEFT_BLIND` 时打印 `sys_tick + L3/R3`
2. `TURN_LEFT_ROTATE -> TRACKING` 时打印全部传感器
3. 再次进入 `TURN_LEFT_BLIND` 时打印与上一次出弯之间的 tick 间隔

只靠这 3 组日志，通常就足够判断是不是“出弯后立刻重入左转”。

## 9. 后续修正方向

后续如果要修，建议优先处理状态机，不要先动 PID。修正方向建议按优先级排序如下：

1. 给左转触发增加释放条件或边沿触发机制。
2. 给出弯后的再次触发增加短暂屏蔽时间。
3. 收紧 `STATE_TURN_LEFT_ROTATE` 的退出条件，避免扫到边线就提前回循迹。
4. 恢复并重新校准最短转向时间判定。
5. 最后再微调 `Car_SetSpeed(-1, 30)` 对应的转向速度参数。

## 10. 总结

当前代码中，“转弯后无法回到直线循迹”的最主要嫌疑并不是程序没有执行 `current_state = STATE_TRACKING`，而是：

- 退出转弯过早；
- 回到循迹后又被立即重新触发进入左转。

因此问题本质上是左转状态机的入弯、出弯、再触发保护三者之间没有形成闭环约束。
